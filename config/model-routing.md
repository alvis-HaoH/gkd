# 模型路由策略

> 这是给主 Claude 看的**跨模型选择偏好**,纯自然语言,可自由编辑。
> 每个模型**能干什么**(能力/禁忌/价格)写在 `models.json` 的 `capabilities`/`avoid_for`/`description`/`pricingKey` 字段里——**本文件不重复那些**,只描述"在多个候选里怎么权衡"。
> 委派时(单次 ask/do 或 workflow 批量),主 Claude 应:**先读 models.json 看每个模型的能力,再按本文件的偏好挑最合适的那个**。

## 选择顺序

1. **用户显式指定优先**:用户传了 `--<modelKey>`,一律服从,不要自作主张分流或"优化"。
2. **硬约束排除**:任务命中某模型的 `avoid_for`(如视觉输入、大批量脏活)就排除它。
3. **能力匹配**:从剩下的候选里,按 `capabilities` 选能胜任的。
4. **能省则省**:多个都能胜任时,默认选**最便宜**的那个(用 `pricingKey` 对应的价格判断,或 description 里的性价比描述)。

## 偏好基调

- **默认倾向省钱**:绝大多数委派(coding、改写、分析、批量脏活)用最便宜的能胜任模型即可。这是 GKD 存在的理由——省 token、省钱。
- **贵模型只在值得时用**:高难推理、brainstorm 发散、交叉 review 等"质量 > 成本"的场景,才升级到最贵档。不要默认用贵的。
- **特殊能力按需切**:任务需要某种特殊能力(如视觉/图片输入)时,切到具备该能力的模型,哪怕它不是最便宜的。

## workflow 批量场景的额外提醒

`/gkd:workflow` 的核心优势是**按子任务挑模型**:同一批 item 里,不同 item 可以走不同模型(视觉 item 一个模型、普通 coding 另一个、难任务再换)。不要无脑给全部 item 套同一个模型——逐 item 判断画像,各派最合适最省的那个。
